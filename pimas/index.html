A personal project — pimas

Let an agent see what happens before it happens.

Every way an AI agent touches software today, it has to do a thing to find out what the thing does — then look again. I’ve been building a reactive engine where the agent runs the exact what-if in a shadow, commits nothing, and leaves the real system untouched.

This runs pimas in your browser — the same speculate the framework ships. The agent’s policy is scripted; the shadow evaluation is real (nothing commits until the final commit).

Why non-committal is the whole point

Observation shouldn’t require mutation.

01Side effects don't fire

A probe isn't free — it can send a request, write a row, trigger a cascade. Speculate recomputes the derived result with no effect flushed. The network, the database, the DOM are never touched.

02The live app never goes wrong

The search runs in a shadow, so nobody watching sees the app thrash, the store never holds garbage between tries, and a crash mid-search can't freeze it in a broken state.

03Planning is counterfactuals

To choose between three moves you must evaluate branches you won't take. If evaluating means committing, you can't compare without entering all three. Speculate weighs them against the same start; the agent commits once.

04Exact, not approximate

Learned agent world-models guess the outcome and drift. pimas re-runs the app's own pure logic against a shadow of the real dependency graph — bit-identical to what committing would have produced.

Why an ordinary stack can’t

It’s cheap only on a graph that already exists.

pimas is a fine-grained reactive engine: state is observable, and it keeps the dependency graph (what derives from what) separate from the values flowing through it. So a hypothetical is just: shadow the values, reuse the real graph read-only, don’t flush the effects, and roll back by throwing the shadow away. The same surface projects onto the browser’s WebMCP tool standard as simulate_* tools an agent just calls.

A virtual-DOM framework has no standing dependency graph to shadow — it would have to re-render and diff, with no guarantee the result is side-effect-free. Committing is how you normally find out what happens; here the agent finds out without it happening — so the UI stops being a surface you poke and becomes a model you query.

“Isn’t this just…?”

Non-committal what-if isn’t new. This combination is.

Speculative execution is a well-worn idea — a database can do it, a spreadsheet can do it. What no one has put together is exact + zero side-effects + on a live reactive UI graph + as a first-class agent tool. Each neighbour gives up one of those:

Optimistic UI / snapshots
commit, then roll back — the side effects already fired.
Agent sandboxes / computer-use
really execute, just in an isolated copy — effects fire there, and it’s a whole environment.
LangGraph time-travel / fork
re-executes nodes — ‘LLM calls, API requests fire again’ (their docs).
Tool approval (human-in-the-loop)
shows the proposed call, not the exact resulting state.
Learned agent world-models
approximate — they guess the outcome and drift.
Datomic with · spreadsheets · PRAXA
real non-committal what-if — but on a DB value, a cell grid, or an analyst’s view; not a live UI an agent queries.

pimas is the one that is all four at once.

Sharpest fit

Agents that plan over state where a wrong commit costs something.

  • →Agentic workflows & ops — preview the exact resulting state, then approve → commit, instead of mutating live systems to see what an action does.
  • →Robotics & physical orchestration — ‘simulate before you act’ as a safety property when the committed action moves something real and irreversible.
  • →Quantitative models — pricing, project economics, risk — the model is already pure, so exact what-if is free and what-if is the whole job.
  • →Tooling for AI agents — give any agent a non-committal preview of a tool's effect before it fires, as a first-class capability.
It’s real code and a reproducible benchmark, not a deck. Building agents that touch live state? Let’s go deep.
